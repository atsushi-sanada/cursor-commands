# pullrequestUpstream

- 現在のブランチのコミットを Upstream/develop ブランチに向けてプルリクエストを作成してください。
- 完了後、プルリクエストのページを開いてください。
- 作業は MCP ＞ CLI ＞ ブラウザの優先度でトライしてください。

## プルリクエストのタイトル（命名ルール）

接頭辞（Type）を統一し、履歴・検索・レビュー・リリースノート生成をしやすくする。

### 基本フォーマット

```text
<Type>: <内容（日本語）>
```

- Type のみ英語、コロン以降は日本語で記述する

### 接頭辞一覧

| Type     | 用途                 | 例                                      |
| -------- | -------------------- | --------------------------------------- |
| Add      | 新機能追加           | `Add: CSV エクスポート機能を追加`       |
| Fix      | バグ修正             | `Fix: 税額計算の誤りを修正`             |
| Update   | 振る舞い変更         | `Update: パスワード検証ルールを変更`   |
| Refactor | 構造改善（仕様変更なし） | `Refactor: 決済ロジックをサービスに分離` |
| Remove   | 削除                 | `Remove: 非推奨 API エンドポイントを削除` |
| Improve  | 改善（体験・性能）   | `Improve: ページ読み込み時間を短縮`     |
| Docs     | ドキュメント         | `Docs: API 仕様書を更新`                |
| Test     | テスト追加・修正     | `Test: ログインの統合テストを追加`     |
| Chore    | 雑務・依存更新       | `Chore: 依存関係を更新`                 |

### 書き方のルール

- コロンの後は半角スペース（`Fix: ○○`）
- Type は命令形で書く（Add / Fix / Update。Added / Fixed は使わない）
- 具体的に書く（✕ `Fix: バグ修正` → ○ `Fix: userId 未定義時の Null エラーを回避`）

### NG 例

- `修正` / `いろいろ対応` / `update` / `対応しました` / `WIP`（何をしたか不明・検索できない）

## プルリクエストのコメント（本文）テンプレート

- 以下のテンプレートの内容で作成してください。
- 見出しの後ろの `（必須）` / `（任意）` は削除してください。
- **本文は実際の改行を使用してください（`\n` などのエスケープシーケンスを文字列として含めないこと）**

### 各セクションの書き方

| セクション   | 目的                                                                 | 読者想定     |
| ------------ | -------------------------------------------------------------------- | ------------ |
| 概要         | 何をしたか・何ができるようになったかを、非エンジニアにもわかる言葉で | 全員         |
| 背景・目的   | なぜこの対応をしたか（課題・きっかけ）                               | レビュアー   |
| 使用方法     | マージ後にユーザーが実際に使う手順・設定・API のみ                   | 利用者       |
| 変更内容     | 技術的な追加・変更の箇条書き                                        | レビュアー   |
| 補足         | 注意点、影響範囲、検証済み事項、関連情報                             | レビュアー   |

**使用方法の判断基準**

- **記載する**：マージ後に利用者が行う操作（設定画面の場所、呼び出し方、API の使い方など）
- **記載しない**：開発時のデバッグ用コマンド、診断用 MCP Query、一時的な確認手順など

```text
## 概要（必須）

- 非エンジニアにもわかる言葉で、何をしたか・何ができるようになったかを 1〜3 行で
- 専門用語は平易に言い換える（例：TerrainData → 地形データ）

## 背景・目的（必須）

- なぜこの対応をしたか（課題・きっかけ・リリース前の洗い出しなど）

## 使用方法（任意）

- マージ後に利用者が実際に使う手順・設定・API のみ記載
- 開発時のデバッグ用・診断用は含めない

## 変更内容（必須）

- どのような機能や処理を追加／変更したか（箇条書き）

## 補足（任意）

- 注意点、影響範囲、検証済み事項、関連情報など（あれば）
```
