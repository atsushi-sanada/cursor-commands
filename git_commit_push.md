# Git コミット・プッシュ実行プロンプト

## 基本方針

- 変更内容を適切にコミットしてプッシュする
- **この指示は今回の依頼内でのみ有効**とし、以降のエージェント／別タスクへは引き継がない
- **現在のチャットで行った修正のみをコミット対象とする**（全ての変更をコミットする場合はユーザーが明示的に指示する）
- コミットメッセージは短く簡潔に、タイトルと本文を分けて箇条書きで記述する

## 安全ルール（差分を破棄しない）

- **ユーザーから明示指示がない限り、差分を破棄する操作は禁止**
  - 禁止例：`git restore .` / `git checkout .` / `git reset --hard` / `git clean -fd` / `git checkout -- <path>` など
- **関係ない差分は「残したまま、触らず、ステージングもしない」**
  - 必要なら「コミット対象を絞る（`git add -- <path>`）」だけで対応する

## 実行手順

1. **変更確認**: `git status --porcelain`で変更状況を確認（ページネーション回避）
2. **コミット対象の確定（破棄しないための確認）**:
   - 「今回コミットするファイル一覧」を提示する（ユーザー指定があればそれを優先）
   - **一覧に含まれない差分は、残したまま未ステージのままにする**
3. **ステージング（コミット対象のみ）**:
   - ユーザー指定がある場合: `git add -- <指定ファイル・ディレクトリ>`
   - 指定が無い場合: **現在のチャットで編集・作成したファイルのみ**を `git add -- <ファイルパス>` で追加
   - ユーザーが「全て」と明示した場合のみ: `git add .`
   - ステージ後の確認: `git diff --cached --name-only`
4. **コミットメッセージ作成**:
   - `GIT_PAGER=cat git diff --cached`で変更内容を確認（ページネーション回避）
   - タイトルと本文を分けて、簡潔に箇条書きでメッセージを作成
5. **コミット**:
   - メッセージ案は提示する（ログとして残す）
   - **差分を破棄する操作が不要で、コミット対象が手順 2 で確定している場合は、承認を待たずに commit → push まで実行**（下の HEREDOC 例を使用）
   - 例外：差分破棄が必要 / 取り込み方針（merge/rebase 等）の判断が必要 / 対象ファイルが曖昧、のいずれかに該当する場合のみユーザー確認を取る
6. **プッシュ**: `git push origin $(git rev-parse --abbrev-ref HEAD)`で実行

### コミット（HEREDOC）例

```bash
git commit -m "$(cat <<'EOF'
<TITLE>

- <BODY>
EOF
)"
```

## 注意事項

- `git diff`はページネーションが起動するため、`GIT_PAGER=cat`を使用するか`--stat`で統計情報のみ確認
- プッシュ前にリモートとの差分がある場合は、まず `git fetch` → `git status -sb` で状況を明確化し、**取り込み方針（merge/rebase）を決めてから**実行する（勝手に `git pull` しない）
